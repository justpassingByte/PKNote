# GTO Abstraction & Card Removal Graceful Fallback

## The Core Concept: Representative Boards
To optimize computation time and storage, our GTO Pipeline groups thousands of different specific flops into 18 generalized "buckets" (e.g., `connected_mid`). For each bucket, the system uses exactly ONE **Representative Board** to run through TexasSolver (e.g., `connected_mid` uses `Ts,9d,8c`).

## The Edge Case: Blocked Hands (Card Removal Effect)
When a user inputs their actual hand and specific board (e.g., Holding `Ts9s` on a `Ks,8s,3c` flop), the hand is completely legal.
However, when the backend queries the database for this bucket (`connected_mid`), the strategy exists ONLY for the representative board `Ts,9d,8c`. 
Since the 10 of Spades (`Ts`) is **already on the representative board**, TexasSolver correctly filters out any `Ts` from the starting hands. 
Thus, the EXACT combination `Ts9s` physically **does not exist** in the resulting DB tree for that bucket.

## The Solution: Graceful Class Fallback
Rather than failing or showing "Hand not specified/null", the system implements a graceful fallback mechanism:
1. **LLM Parsing (`hero_hand_class`)**: The prompt is configured to not only extract the exact hand (e.g. `Ts9s`) but also analyze the board context and classify the hand (e.g. `hero_hand_class: "flush_draw"`).
2. **Backend Interception**: In `GtoController.ts`, when the backend searches for `Ts9s` and gets a `null` (due to card removal blockers from the representative board), it triggers the fallback.
3. **Averaging Strategy**: The backend utilizes the pre-aggregated `classSummary` generated by TexasSolver. It pulls the average check/bet/fold frequencies for the entire `flush_draw` class within that specific spot.
4. **Faking the Hero Combo**: The backend returns the `classSummary` frequencies back to the frontend, cleverly disguised under the user's original `Ts9s` identifier.
